# 生图 Skill 设计

## 背景

用户希望基于现有图片生成接口封装一个可复用的 Claude skill，后续可以直接通过该 skill 发起生图请求。接口地址与 API Key 不应写入仓库，而应保存在项目本地环境变量文件中。

当前接口文档显示该服务适合封装为单职责 skill：请求路径固定、鉴权清晰、参数较集中、返回结果简单，并且错误码覆盖较完整。

## 目标

1. 在仓库内提供一个可复用的本地 Claude 生图 skill
2. 通过项目根目录 `.env.local` 管理接口地址与 API Key
3. 第一版优先支持文生图
4. 成功时返回图片 URL，失败时返回清晰错误信息

## 方案选择

### 方案一：项目级 Claude Skill + `.env.local`（最终选择）

- 在仓库内新增项目级 skill
- skill 调用本地脚本完成请求
- 配置从 `.env.local` 读取

优点：结构清晰、仓库内自洽、便于未来继续扩展。

### 方案二：只有脚本，没有 skill

不符合用户“以后让我直接用它生成图片”的使用方式，因此不采用。

### 方案三：首发就支持完整图生图 / 批量 / 多模式

功能过重，不符合当前 YAGNI，因此不采用。

## 最终设计

### 1. Skill 结构

- 新增项目级 skill，暂定名为 `image-generator`
- skill 的职责是：
  1. 接收用户自然语言生图请求
  2. 整理请求参数
  3. 调用本地脚本发起 API 请求
  4. 返回图片 URL 或清晰错误

### 2. 配置方式

在项目根目录 `.env.local` 中读取以下变量：

- `IMAGE_API_BASE_URL`
- `IMAGE_API_KEY`

可选新增 `.env.example`，仅保留变量名示例，不写真实值。

### 3. 首发参数范围

第一版默认支持：

- `prompt`（必填）
- `size`（可选）
- `model`（可选）
- `timeout_sec`（可选）

第一版优先做文生图，不把图生图作为首发能力。

### 4. 请求与返回

- 请求方式：`POST /v1/images/generations`
- 认证方式：Bearer Token
- 返回解析：提取 `data[].b64_json`
- 注意：虽然字段名为 `b64_json`，但接口文档说明实际返回的是图片 URL，因此应按 URL 处理。

### 5. 错误处理

需要对以下场景返回清晰错误：

- 缺少 `.env.local`
- 缺少 `IMAGE_API_BASE_URL` 或 `IMAGE_API_KEY`
- `401` 未认证
- `429` 限流 / 排队超时
- `503` 上游账户不可用或 token 失效
- `504` 任务超时
- 其他非预期错误

### 6. 扩展边界

第一版不做：

- 图生图完整工作流
- 批量生成
- 图片下载与本地管理系统
- 通用 SDK 抽象层

后续如有需要，再增加参考图 `image[]`、更多模型参数、保存文件能力等。

## 验收标准

- 仓库内存在可调用的生图 skill
- skill 能从 `.env.local` 正确读取配置
- 给定 prompt 时可成功返回图片 URL
- 配置错误和接口错误时，提示清晰
- 真实密钥不进入仓库

## 原则应用

- **KISS**：以“生成图片”作为单一职责，不扩展成通用平台
- **YAGNI**：首发只做文生图必需路径
- **DRY**：把请求逻辑集中在本地脚本，避免 skill 内重复处理细节
- **SOLID**：skill 负责交互，脚本负责请求，配置由 env 管理，职责分离清晰
