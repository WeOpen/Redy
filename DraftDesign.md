# WeOpen 吉祥物 Redy 正式设计方案

**项目名称：** WeOpen 品牌吉祥物与视觉识别系统设计
**吉祥物名称：** Redy
**适用范围：** 品牌传播 / 官网与 GitHub / 社区运营 / CLI 工具 / 周边物料 / 动效与表情体系

---

# 1. 项目背景与设计目标

## 1.1 项目背景

WeOpen 作为一个具有开源属性与技术社区气质的组织，需要一套具有长期识别性、扩展性与情感温度的品牌视觉符号，用于统一组织形象、强化社区记忆点，并在 GitHub、官网、CLI、社群运营和周边物料等多个场景中形成稳定输出。

在视觉表达上，WeOpen 希望兼具两种气质：
一是温润、柔和、有陪伴感的手绘氛围；
二是简洁、模块化、具有 Git 与开源文化隐喻的极客审美。

因此，本方案以红熊猫吉祥物 **Redy** 为核心，构建一套兼具品牌人格、视觉规范、交互语言与衍生能力的完整系统。

## 1.2 设计目标

本项目的核心目标包括：

1. 建立 WeOpen 可长期使用的品牌吉祥物形象；
2. 统一组织在不同媒介中的视觉识别；
3. 为设计、开发、运营和物料制作提供标准化执行依据；
4. 让吉祥物不仅是“插画形象”，更成为具有品牌叙事能力的“生态角色”；
5. 在温暖亲和与技术专业之间建立清晰平衡。

# 2. 吉祥物品牌定位

## 2.1 吉祥物命名

**Redy**
命名由 **Red + Ready** 组合而来，既对应红熊猫的核心色彩，也传达 “准备就绪、可靠、可协作” 的品牌气质。

## 2.2 核心角色定位

Redy 不只是 WeOpen 的卡通形象，而是组织的品牌人格化代表。其正式定位为：

**WeOpen 开源生态维护者 / 代码园丁 / 分支探险家**

这一定位用于统一所有视觉与文案解释体系：

* **红熊猫**：敏捷、专注、友好、温和而可靠
* **护目镜**：深入代码细节、调试与观察视角
* **工具腰带**：技术工具箱与项目生态能力
* **Git 分支树 / 开源森林**：协作式生长、版本演化、开源共建
* **Cherry-pick 行为**：精选优质贡献、组织有价值的代码成果

## 2.3 品牌人格关键词

Redy 的品牌人格建议统一使用以下关键词描述：

* 冷静
* 极客
* 温和
* 可靠
* 专注
* 审慎
* 观察者
* 维护者
* 共建者

## 2.4 品牌语义主张

围绕 Redy，可统一形成 WeOpen 的品牌表达：

* 技术可以是温暖的
* 协作不是冷冰冰的流程，而是共同生长的生态
* 开源不是无序堆叠，而是被精心维护的森林
* WeOpen 不只是构建工具，也在挑选、整理与培育更好的开源体验

# 4. 视觉概念与风格策略

## 4.1 总体艺术方向

本方案采用双重融合风格：

### 4.1.1 主风格：温润手绘感

参考方向为具有工作室氛围的手绘插画风格，强调：

* 柔边棕褐线条
* 低饱和暖色系
* 漫反射光影
* 水彩薄涂与毛发肌理
* 温和、安静、具有陪伴感的画面氛围

### 4.1.2 辅风格：极客像素感

作为辅助层嵌入，不可喧宾夺主。主要用于：

* 护目镜反光
* 工具挂件
* Git 分支符号
* 终端光标元素
* 尾巴与配件局部像素纹理

整体原则是：
**主体温润，细节极客；整体柔和，局部精确。**

# 5. Redy 角色设计规范

## 5.1 基本设定

* **物种：** 红熊猫
* **体态：** 圆润、轻盈、蓬松、敏捷
* **表情基调：** 冷静、专注、略带观察感，不做过度夸张的卡通表情
* **动作气质：** 克制、安静、有判断力，不做喧闹型 mascot

## 5.2 标志性特征

### 护目镜

护目镜是 Redy 的第一识别点，必须在多数版本中保留。
要求如下：

* 复古飞行护目镜造型
* 镜框偏浅灰或炭灰
* 镜片可加入轻微 Git 绿发光
* 可在镜片中嵌入终端光标 `>_` 或 Git 分支反光细节
* 既有“观察”感，也有“技术工具”感

### 工具腰带

腰带用于表达 Redy 的“维护者 / 园丁 / 工具型角色”身份。

标准挂件配置：

* **WeMail 挂件**：以信封、蜡封、Cloudflare 云形轮廓或临时邮箱标识为灵感，象征 WeOpen 当前已落地的邮件产品能力
* **WeGit 挂件**：以原生窗口轮廓、分支线或提交节点为灵感，象征 Git 协作、版本演化与未来的 macOS 原生 Git 客户端方向
* **WeOpen 挂件**：以控制台面板、仪表盘框架或模块化入口为灵感，象征个人综合管理平台与 WeOpen 生态主入口

### 尾巴

尾巴是 Redy 的第二识别特征。
要求蓬松、层次感明显，并可承担更多图形抽象能力：

* 在角色本体中表现毛发柔软感
* 在系统图形中抽象为“像素笔刷”或动态拖尾
* 在动效中承担情绪反馈和状态反馈作用

# 6. 标准视觉规范

## 6.1 标准色板

以下颜色建议作为官方统一品牌色板使用：

| 颜色名称  |      色值 | 用途                |
| ----- | ------: | ----------------- |
| 主体锈红  | #C14B30 | Redy 主体毛发、主视觉主色   |
| 奶油白   | #F6F1E9 | 胸口、腹部、耳尖、浅色背景     |
| 深炭灰   | #2D2A2E | 腰带、挂件、文字、结构细节     |
| Git 绿 | #2CC68E | 护目镜微光、Git 符号、强调反馈 |
| 暖米背景  | #F8F5F0 | 网页与海报主背景          |
| 柔边棕褐  | #8B6E54 | 轮廓线条、结构边线         |

## 6.2 用色原则

1. 主视觉色彩以锈红、奶油白、暖米为主体；
2. 深炭灰替代纯黑，避免气质过冷；
3. Git 绿只能作为强调色，面积不宜过大；
4. 禁止使用高饱和荧光色破坏整体温润基调；
5. 允许在大幅场景插画中使用柔和暖光渐层，但主体角色不建议使用强烈赛博渐变。

## 6.3 三项目配色与强调色规范

在母品牌统一色板基础上，WeMail、WeGit、WeOpen 三个项目不应分别建立完整独立色板，而应仅增加轻量级辅助强调色，用于局部识别与场景区分。

### 6.3.1 WeMail

* 建议强调为云雾浅蓝灰、冷白蓝光或略带蜡封质感的柔和红棕点缀
* 适合用于信封描边、邮件卡片高光、蜡封细节、Cloudflare 语义提示等局部位置
* 面积应小、亮度应轻，重点表达“通信、临时、私密、可信递送”的气质

### 6.3.2 WeGit

* 建议继续以 Git 绿作为第一识别强调色，不额外发明新的主识别颜色
* 适合用于 branch graph、commit node、窗口状态点、操作反馈高光等技术结构元素
* 应强化其版本控制、分支协作与原生客户端的技术属性

### 6.3.3 WeOpen

* 建议以暖灰米色系与少量锈红延展作为辅助强调
* 适合用于控制台边框、模块卡片层级、平台骨架与统一入口结构
* 应优先体现“平台、组织、总入口、模块编排”而非单点工具能力

### 6.3.4 使用限制

* 项目强调色只能作为辅助，不能压过 Redy 本体主色
* 同一画面最多只允许一个项目强调色成为局部视觉焦点
* 三项目图标同屏时，应先保持统一材质与线条语言，再进行色彩区分
* WeOpen 作为平台级项目，其色彩表达应更稳、更中性，不应比 WeMail 与 WeGit 更跳脱

## 6.4 线条与质感规范

* 轮廓采用柔边棕褐，不使用纯黑硬边
* 线条允许轻微手绘抖动感
* 毛发表现采用薄水彩叠加或浅纹理刷
* 光影以漫反射和暖色边缘柔光为主
* 避免高反差、高金属质感和过度赛博化效果

# 7. 标准造型系统

## 7.1 造型 A：Logo / 极简头像

**用途：** favicon、社群头像、徽章、组织图标、小尺寸识别

设计要求：

* 采用侧脸或探头构图
* 护目镜为识别核心
* 可在 16×16px 下保持辨识
* 需同时输出单色版、双色版和反白版

## 7.2 造型 B：标准全身立绘

**用途：** 官网、海报、GitHub、运营主视觉

设计要求：

* 蹲坐或轻前倾观察姿态
* 护目镜、尾巴、腰带挂件齐全
* 保持角色稳定、可复用、可换姿态扩展

## 7.3 造型 C：园丁场景版

**用途：** Banner、开源社区主视觉、活动页头图

设计要求：

* Redy 坐在 Git 分支树枝上
* 手持放大镜进行 cherry-pick
* 周围悬挂或环绕三类项目识别元素：WeMail、WeGit、WeOpen
* 背景为“开源森林”概念图景，并以不同项目模块作为生态节点

## 7.4 造型 D：Q 版迷你版

**用途：** 404 页、加载动画、弹窗提示、社群贴纸

设计要求：

* 大头小身
* 动作简单明确
* 适合情绪化表达与小尺寸场景传播

# 8. 三视图描述规范

为保障不同画师和供应商执行统一，Redy 需具备标准三视图描述。

## 8.1 正面视图

* 正蹲坐，身体整体对称
* 头部圆润，耳朵外轮廓明显
* 护目镜居中覆盖眼部区域
* 腹部与胸口奶油白形成视觉中心
* 工具腰带在正面可见，挂件左右分布明确
* 尾巴在身体后方展开，与身体宽度基本相当

## 8.2 侧面视图

* 身体轻微前倾，体现观察与专注
* 鼻尖、耳朵与护目镜轮廓清晰
* 腰带与挂件有前后层次
* 尾巴自然弯曲，贴合身体侧方
* 护目镜高光与 Git 绿反光保留

## 8.3 背面视图

* 背部圆润，毛发方向自然向后延展
* 护目镜绑带从后脑绕过
* 工具腰带背面保留基础搭扣结构
* 尾巴蓬松扇形展开，可带轻微奶油白尾端细节
* 背面不出现多余复杂装饰，以便供应商简化制作

# 9. 图形语言系统

除了角色本体，Redy 需要配套的图形语言，以支持 UI、海报、README、贴纸与动效的统一。

## 9.1 Git 分支线

作为 WeOpen 品牌的基础图形骨架，可用于：

* 页面分割线
* 背景结构纹理
* Loading 动画路径
* Banner 中的视觉引导线

要求简洁、轻量，不宜过度密集。

## 9.2 项目识别符号系统

建议建立一组稳定的项目识别图形，与 WeOpen 现有和规划中的三个核心项目直接关联：

| 项目      | 建议形态                    | 含义 |
| ------- | ----------------------- | ---- |
| WeMail  | 信封、蜡封、收件箱、云轮廓、一次性邮件卡片结构 | 基于 Cloudflare 的临时邮箱能力与轻量通信入口，并强化“可信投递”与“短时私密通信”的气质 |
| WeGit   | 原生窗口轮廓、分支线、提交节点、仓库标识    | macOS 原生 Git 客户端方向、版本管理与协作流程 |
| WeOpen  | 控制台面板、模块卡片、仪表盘骨架、统一入口框架 | 开源个人综合管理平台、生态聚合与多模块组织能力 |

这些元素可进一步转化为：

* 项目 Icon
* 项目 Badge
* README 状态图形
* Banner 中的生态节点
* 运营贴纸与说明性插画资产

## 9.3 项目视觉映射规范

为保证三项目在同一母品牌下既统一又可区分，建议建立一套明确的项目视觉映射规范。

### 9.3.1 母品牌层

* Redy 只代表 WeOpen 母品牌人格，不单独代表某一个项目
* WeMail、WeGit、WeOpen 三个项目不再各自发展独立 mascot
* 三项目必须共享统一的线条风格、材质气质、场景骨架与品牌色板基础
* 所有项目视觉都应优先让用户识别出“WeOpen 家族”，其次才是具体项目差异

### 9.3.2 项目识别层

* **WeMail**：以信封、蜡封、收件箱、临时邮件卡片、云轮廓等元素作为主识别符号，蜡封用于强化“可信、私密、被正式递送”的情绪印象
* **WeGit**：以 branch graph、commit node、仓库标识、macOS 原生窗口轮廓作为主识别符号
* **WeOpen**：以模块卡片、dashboard 骨架、控制台面板、统一入口框架作为主识别符号
* 三项目均可拥有各自的图标组合，但不得脱离 Redy 所在的母系统图形语言

### 9.3.3 场景映射层

* 在官网主视觉中，三项目应作为开源生态节点出现在 Redy 所维护的系统场景中
* 在 GitHub Header 中，三项目可沿 Git 分支线或模块结构展开，作为生态模块而非独立角色出现
* 在 README、Badge、贴纸、社群说明图中，可将三项目识别元素拆解为独立 icon 使用
* 在介绍 WeOpen 组织能力时，应优先呈现三项目之间的协同关系，而非单独强调某一个产品

### 9.3.4 限制规则

* 不允许将 WeMail、WeGit、WeOpen 分别画成三个不同动物角色
* 不允许为三个项目建立风格差异过大的独立插画体系
* 不允许项目强调色压过母品牌主色与 Redy 的主体识别
* WeOpen 作为平台级项目，在系统总览场景中的视觉权重应略高于单点工具型项目

## 9.4 像素笔刷尾巴

将 Redy 尾巴抽象为品牌辅助图形，用于：

* hover 动效拖尾
* Banner 装饰纹理
* 社群表情背景元素
* CLI ASCII 轮廓灵感来源

# 10. Figma 组件与设计资产规范

## 10.1 设计原则

* 可复用
* 可缩放
* 多场景适配
* 保持气质统一
* 降低后续二次修改成本

## 10.2 推荐基础资产尺寸

| 组件   |         建议尺寸 | 用途           |
| ---- | -----------: | ------------ |
| 标准立绘 |  800×1000 px | 海报、官网、主视觉    |
| 圆形头像 |   512×512 px | 社群头像、徽章      |
| 极简图标 |   128×128 px | icon、favicon |
| Q版组件 |   400×400 px | 提示弹窗、贴纸      |
| 园丁场景 | 1920×1080 px | Banner、官网头图  |

## 10.3 组件间距规范

* 网页中 Redy 与文字最小安全距离建议不小于 24px
* 小尺寸贴纸与徽章应用中最小安全距离不小于 16px
* 图标版不建议加入多余背景装饰，留白优先

## 10.4 图层建议

1. 背景层
2. 主体层
3. 细节层
4. 光影层
5. 极客点缀层

极客点缀层透明度不宜过高，避免破坏整体手绘氛围。

# 11. 行为系统与状态语法

为保证 Redy 在产品体验中的一致性，建议建立行为语法。

| 场景  | Redy 行为      | 含义         |
| --- | ------------ | ---------- |
| 初始化 | 戴护目镜检查       | 环境检测 / 准备中 |
| 成功  | 尾巴轻摆 / 镜片闪烁  | 操作成功       |
| 错误  | 放大镜检查 / 困惑歪头 | 异常 / 未找到   |
| 更新  | 点亮项目节点 / 举小旗  | 新版本发布或项目进展更新 |
| 分支  | 观察树枝 / 分支点亮  | Git 行为与协作  |
| 合并  | 点头 / 护目镜亮三次  | Merge 完成   |

这一系统可直接映射到 UI、CLI 和运营插画。

# 12. 动效设计规范

## 12.1 总体原则

Redy 的动效应保持“柔和、克制、可爱但不闹腾”的节奏。
不建议使用强烈弹跳、夸张拉伸、过多粒子喷发等偏游戏化表现。

## 12.2 护目镜动效

* 常态：保留弱反光与微弱 Git 绿点
* Hover：镜片绿光缓慢流动，时长约 1.5 秒
* 成功反馈：闪烁 3 次后恢复常态

## 12.3 尾巴动效

* 待机：轻微缓慢左右摆动
* 开心：摆动幅度增加
* 困惑：尾巴下垂，停止摆动
* 专注：整体静止，仅尾尖微颤

## 12.4 眼神跟随

适用于官网或 Banner：

* 幅度小
* 延迟柔和
* 不做夸张追踪
* 保留“观察者”人设气质

## 12.5 Loading 动效

建议使用 Q 版 Redy：

* 尾巴绕体缓慢转动
* 镜片内部 Git 绿线流动
* 身体轻微浮动
* 加载完成后镜片闪一下

# 13. 应用场景规范

## 13.1 组织 Logo

Logo 采用极简侧脸结构，Redy 从圆形负空间中探出头部。
适合 favicon、GitHub organization icon、徽章与小程序入口等高频识别场景。

## 13.2 GitHub Header / Banner

推荐标准布局：

* 左侧：Redy 半身或园丁形态
* 中部：Git 分支结构延展
* 右侧：WeOpen 标志文字与 slogan

建议标语：
**Build. Pick. Evolve.**
或
**Open source, warm and ready.**

## 13.3 项目协同主视觉

采用“开源森林 / 园丁概念”：

* Redy 在 Git 分支树上 cherry-pick
* 项目成果以 WeMail、WeGit、WeOpen 三个项目识别元素呈现
* 代码块、项目模块卡片与像素叶片共同构成场景语言

此方案适合：

* 社区介绍页
* 开源活动 KV
* 贡献指南封面
* 年度总结海报

## 13.4 三项目图标 / Badge / README 头图使用规范

为保证 WeMail、WeGit、WeOpen 在 GitHub、README 与项目主页中保持统一品牌感，建议建立以下使用规范。

### 13.4.1 项目图标

* 三项目图标应共享统一线条粗细、圆角语言、材质氛围与留白结构
* 图标中可出现项目主识别符号，但不应直接复制复杂场景插画
* **WeMail** 图标优先使用信封、蜡封、云轮廓等组合
* **WeGit** 图标优先使用分支线、提交节点、原生窗口轮廓等组合
* **WeOpen** 图标优先使用模块卡片、控制台骨架、统一入口框架等组合
* 项目图标建议优先输出 SVG 与 128×128、256×256 PNG 版本

### 13.4.2 项目 Badge

* Badge 应基于项目图标做最小化抽象，不直接使用完整 Redy 角色
* Badge 更适合承担“项目识别”和“状态识别”，不承担复杂叙事功能
* 当多个 Badge 同时出现时，优先通过图形结构区分项目，再通过局部强调色补充识别
* 同组 Badge 应保持统一边框逻辑、统一背景明度与统一阴影强度

### 13.4.3 README 头图

* README 头图应采用“母品牌统一 + 项目局部识别”的结构，不建议每个项目都做完全不同的视觉世界
* 推荐布局为：左侧或中央为 Redy / 母品牌识别，中部为项目主符号与标题，背景使用 Git 分支线、模块结构或轻量生态纹理
* WeMail README 头图可强化信封、蜡封、临时邮件卡片与轻云感元素
* WeGit README 头图可强化 branch graph、提交节点、原生窗口与桌面客户端气质
* WeOpen README 头图可强化控制台框架、模块卡片、平台入口与聚合管理气质
* 所有 README 头图都应保留统一的暖米背景体系或其等价浅色变体，避免三个项目各自漂移到完全不同的色域

### 13.4.4 使用限制

* 不允许把 README 头图做成三个完全不同画风的独立品牌首页
* 不允许在小尺寸 Badge 中塞入 Redy 全身形象，避免识别噪音
* 不允许项目图标使用与母品牌冲突的高饱和色作为主体色
* 同一项目在 GitHub、官网与社群平台上的图标主结构应保持一致，不应频繁更换识别主体

## 13.5 404 / 异常页面

推荐使用 Q 版困惑 Redy：

* 低头寻找
* 手持放大镜
* 背景轻量，不抢文案信息

推荐文案：

* **Branch not found.**
* **Redy is searching the open forest...**
* **Looks like this path hasn’t been merged yet.**

## 13.6 CLI 欢迎界面

建议保留 Redy 的极客仪式感。正式推荐文案如下：

## 13.7 更新日志 / 贡献指南

* Merge 完成：Redy 点头

* Bug 修复：Redy 拿小刷子清理代码碎屑
* Release 发布：Redy 点亮对应项目节点或举起版本旗标

# 18. 实施优先级建议

为提高落地效率，建议将 Redy 项目按“先建立母品牌基线，再落地三项目数字资产，最后扩展运营与周边”的顺序推进。

## 18.1 第一阶段：品牌基础建立

**目标：** 建立 WeOpen / Redy 母品牌的最小可用识别系统。

**优先产物：**

* 完成 Logo 极简版
* 完成标准立绘
* 完成色板、线条、三视图
* 完成 GitHub Header 首版
* 完成品牌基础指南第一页版本

**完成标准：**

* Redy 的核心识别元素固定下来
* 母品牌色板与三项目配色规则可复用
* GitHub 与官网至少拥有一个可直接上线的主视觉版本

## 18.2 第二阶段：三项目数字场景落地

**目标：** 让 WeMail、WeGit、WeOpen 三个项目在 GitHub、README 与官网场景中形成统一但可区分的视觉输出。

**优先产物：**

* WeMail 项目图标、Badge、README 头图
* WeGit 项目图标、Badge、README 头图
* WeOpen 项目图标、Badge、README 头图
* 官网 Hero 视觉
* 404 页面
* CLI 欢迎输出

**执行顺序建议：**

1. 先完成三项目图标
2. 再完成三项目 Badge
3. 再基于图标系统延展 README 头图
4. 最后统一输出官网与 GitHub 场景图

**完成标准：**

* 三项目在图标层、Badge 层、README 层均已形成可直接使用的资产
* 三项目可被清晰区分，但母品牌统一性未被破坏
* GitHub 仓库首页与官网展示已具备一致的品牌观感

## 18.3 第三阶段：运营与社区传播扩展

**目标：** 基于既有品牌系统扩展传播资产与轻量互动场景。

**优先产物：**

* 海报 KV
* 表情与贴纸初版
* 社区活动物料
* README 动态小交互
* 周边打样

**完成标准：**

* Redy 具备从品牌识别到社区传播的连续表达能力
* 三项目资产可以进入对外宣传、社区活动与社媒传播场景
* 贴纸、海报、周边与线上主视觉风格统一

## 18.4 第四阶段：动效与品牌宇宙扩展

**目标：** 让 Redy 从静态品牌资产升级为具备动效与角色生态扩展能力的系统。

**优先产物：**

* 动效系统
* Loading 与状态反馈动效
* Banner 轻交互动效
* Redy 宇宙角色延展（Dev / AI / Ops）
* Redy Forest System 的长期扩展设定

**完成标准：**

* Redy 在官网、CLI、社群传播中具备统一的动态语法
* 扩展角色不会破坏现有母品牌识别
* 动效素材与静态素材可共用同一套视觉规则

## 18.5 推荐执行节奏

如果按小团队推进，建议采用如下顺序：

1. 先完成母品牌识别资产
2. 紧接着完成三项目图标与 Badge
3. 再统一完成 README 头图与 GitHub Header
4. 然后落地官网 Hero、404、CLI 等数字场景
5. 最后再投入贴纸、动效、海报与周边

这一顺序可以保证最先产出的资产就能被 GitHub、官网与文档系统立即使用，同时避免一开始就投入大量运营扩展物料，导致主识别尚未收敛。

# 19. 执行分工建议

为了让 Redy 方案能够高效落地，建议按角色能力进行协作拆分，而不是按文件类型机械拆分。

## 19.1 品牌 / 视觉主设计

**职责重点：**

* 最终确认 Redy 母品牌形象
* 确认 Logo、主色板、三项目视觉映射关系
* 审核 WeMail、WeGit、WeOpen 是否仍属于同一母系统
* 对所有高曝光场景做最终视觉把关

**优先交付：**

* Redy 主视觉定稿
* 母品牌规范页
* 三项目视觉映射总表
* GitHub Header 与官网 Hero 总方向

## 19.2 插画 / 吉祥物设计

**职责重点：**

* 输出 Redy 标准立绘、Q 版、场景版与贴纸版

* 负责 WeMail 蜡封元素、WeGit 分支结构、WeOpen 平台骨架在插画中的具体表达

**优先交付：**

* 标准立绘
* 园丁场景版
* 404 困惑版
* 表情 / 贴纸初版

## 19.3 图标 / 品牌资产设计

**职责重点：**

* 输出 WeMail、WeGit、WeOpen 三项目图标
* 输出 Badge、README 头图、最小化识别版本
* 负责 SVG / PNG / Figma 组件化整理

**优先交付：**

* 三项目主图标
* 三项目极简图标
* 三项目 Badge
* 三项目 README 头图

## 19.4 前端 / 数字场景落地

**职责重点：**

* 将品牌资产接入 GitHub、官网、README、CLI 等数字表面
* 落地官网 Hero、404、Banner、加载态等前端展示场景
* 维护图像资源命名、导出倍率、暗浅背景适配与开发接入规范

**优先交付：**

* 官网 Hero 首版
* 404 页面视觉接入
* CLI 欢迎输出接入
* GitHub / README 展示资源替换

## 19.5 动效 / 传播物料设计

**职责重点：**

* 基于既有静态资产扩展 Loading、状态反馈、轻交互动效
* 输出海报、活动 KV、传播贴纸、社群运营素材
* 保证动态表现不破坏 Redy 原有气质

**优先交付：**

* Loading 动效
* 成功 / 错误反馈动效
* 活动海报 KV
* 社群贴纸动图版

## 19.6 推荐协作顺序

建议协作顺序如下：

1. 品牌 / 视觉主设计先锁定母品牌基线
2. 插画设计与图标设计并行推进
3. 图标系统稳定后，前端开始接入官网、README 与 GitHub 场景
4. 静态资产稳定后，再进入动效与运营传播物料扩展

这样可以避免前端过早接入未定稿素材，也能避免插画与图标系统分头失控。

# 21. 最终交付摘要版

以下内容可作为发给设计师、品牌负责人、前端同学或协作团队的简版执行摘要。

## 21.1 项目目标

为 WeOpen 建立一个可长期使用的品牌吉祥物与视觉识别系统，以 Redy 为核心，统一 GitHub、官网、README、CLI、社群传播与周边物料的视觉表达。

## 21.2 核心原则

* Redy 是 WeOpen 母品牌吉祥物，不为单个项目独占
* WeMail、WeGit、WeOpen 三项目采用“统一母体，局部区分”的视觉策略
* 不再使用水果符号系统，改为直接绑定三项目识别元素
* WeMail 加入蜡封元素，强化“可信、私密、正式递送”的气质
* 所有项目资产应先保证母品牌统一性，再体现项目差异

## 21.3 三项目识别重点

* **WeMail**：信封、蜡封、收件箱、临时邮件卡片、云轮廓
* **WeGit**：branch graph、commit node、仓库标识、macOS 原生窗口
* **WeOpen**：模块卡片、控制台面板、dashboard 骨架、统一入口框架

## 21.4 第一批必须产出

* Redy 主 Logo
* Redy 极简头像
* Redy 标准立绘
* GitHub Header 首版
* WeMail / WeGit / WeOpen 三项目图标
* 三项目 Badge
* 三项目 README 头图
* 色板 / Token / 使用规范文件

## 21.5 推荐执行顺序

1. 先锁定母品牌基线
2. 再做三项目图标与 Badge
3. 再做 README 头图与 GitHub Header
4. 然后接入官网 Hero、404、CLI 等数字场景
5. 最后扩展贴纸、动效、海报与周边

## 21.6 协作分工

* 品牌 / 视觉主设计：负责母品牌基线与总审美把控
* 插画设计：负责 Redy 本体、场景版、贴纸版
* 图标资产设计：负责三项目图标、Badge、README 头图
* 前端：负责官网、README、GitHub、CLI 等数字场景接入
* 动效 / 传播设计：负责 Loading、状态反馈、活动 KV、社群传播物料

## 21.7 验收标准

* 三项目能一眼区分，但仍明显属于 WeOpen 同一品牌系统
* GitHub、官网、README 至少已有一套可直接上线的完整视觉资产
* SVG、PNG、Figma 源文件与 Token 文件齐备
* 视觉规范、导出文件与实际使用场景保持一致

# 22. 结论

本方案以 Redy 为核心，不仅建立了一个具有高辨识度的红熊猫吉祥物形象，也进一步构建了 WeOpen 可持续使用的品牌视觉系统。
它兼顾了温润与专业、亲和与技术、情绪表达与系统规范，既适合当前阶段快速落地，也具备未来扩展为完整品牌宇宙的基础。

Redy 的价值，不只是“一个可爱的 mascot”，而是 WeOpen 对外表达自身文化的一种方式：
**温和看待技术，严谨对待代码；让开源不只是协作，更是一片持续生长的森林。**
